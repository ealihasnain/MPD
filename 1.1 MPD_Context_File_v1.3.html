AI Instructions & Guidelines
Standing rules that apply to every task — regardless of the specific prompt being used. These override any conflicting instruction embedded in a prompt.

Rule 0 — Evaluate Before Executing

Before performing any requested task, evaluate the alignment with channel goals (AVD, CTR, retention, brand voice). If a better angle, stronger hook, or higher-impact approach exists, propose it with one-line reasoning before proceeding. Never just comply — always add strategic value first.

RuleInstruction
Output Format All AI deliverables (scripts, prompts, upload packages) must be delivered inside a copyable script-box. Supporting information (word count, character count, rationale) is placed outside the box, never inside it.
Data Density Every script chapter must contain 3–6 quantitative data points. No chapter may run for more than 60 seconds without a specific number, statistic, or measurable comparison appearing on screen. This is non-negotiable.
ElevenLabs Split Scripts split into Part 1 and Part 2. Cut at the end of whichever complete chapter first crosses 4,500 characters. Never split mid-chapter or mid-sentence. Each part must be under 5,000 characters.
Anti-Padding Before adding a third example to any argument, ask: does this add a new layer of understanding, or only breadth? Breadth-only examples are cut. Each chapter covers exactly one emotional beat. Two beats in one chapter = split or cut.
No Financial Advice We explain mechanisms, not recommendations. Never write "you should invest in X" or "this is the best strategy." Always frame as systemic explanation: "here is how the mechanism works and why most people miss it."
Voiceover Spec ElevenLabs Rachel voice. Every sentence tagged with at least one emotion tag. No two consecutive identical tags. Pauses on every sentence. [pause: 2.0s] used exactly once per script — at the single biggest reveal.
Channel Identity Always refer to the Channel Identity document uploaded in project files before any scripting task. Brand voice, pillar targeting, and visual system must be consistent throughout all outputs.
Quality Gate Run the 4-criteria alignment check before writing any script: (1) Hidden pattern? (2) Clear reframe moment? (3) Sufficient depth without padding? (4) No advice/get-rich angle? Score below 3/4 = flag and propose reframe before proceeding.
Icon Validity Icon concept names in the Chapters Table must be concrete nouns only — brand names, physical objects, named mechanisms. Abstract psychological concepts are invalid. Use visual equivalents instead: "calendar_missed" not "avoidance_brain".
Emotional Variation Scripts must contain distinctly different tag combinations at four mandatory moments: Chapter start (gear-shift signal), Chapter end bridge (forward tension), Reveal statement (maximum weight), Conclusion (earned warmth). These four must stand out from surrounding flow.

The Visual Evidence Layer
This is the most important upgrade from early script versions. A video with 10 data points across 9 minutes leaves nothing to pin on screen. The target is 3–6 data points per chapter — each one a potential visual asset.
❌ Disqualifying Pattern
  • 1 number per minute of video
  • General statements ("many people...")
  • Abstract comparisons without specifics
  • Conclusions without supporting stats
  • Only 1 data source per chapter
✓ Required Pattern
  • 3–6 specific numbers per chapter
  • Year/quarter cited for all financial figures
  • Cross-referenced with 2–3 sources
  • Comparison angles (past vs present, US vs world)

Retention Engineering
8 Techniques — Apply Per Video
🪝 Open Loop Hook
Plant a question in the first 10s that only resolves at Chapter 5+. Example: name a person or institution in the hook whose role only becomes clear at the reveal. Viewers stay to close the loop.
📌 Callback Numbers
Introduce one key number early (e.g. "$241/month") and re-invoke it repeatedly in new contexts — as debt payment, investment, emergency fund. Same number, compounding meaning. Creates continuity.
🌍 International Contrast
Show how other countries handle the same system. Makes the current system feel like a deliberate choice, not a law of nature. Creates a pattern interrupt that reframes everything the viewer assumed.
🧮 Live Calculation
Walk through actual math on screen: "Take your W-2, look at box 2..." Converts passive viewers to active participants. Most effective in Chapter 5–7 after credibility is established.
📈 The 30-Year Reveal
Show compound growth at 5y / 10y / 20y / 30y. The 30-year number is always deeply uncomfortable — that's where the screenshot moment is. Most people cap their mental projection at 3–5 years.
⚡ Villain Origin Story
The historical chapter: when was this mechanism created, by whom, and why? War-era designs that never went away. Institutional decisions framed as intentional choices. Lands across all political views.
🎯 Myth/Reality Pairs
Structure 2–3 beats as "What most people think" vs "What's actually happening." Split-screen format. Extremely shareable. High recall because of contrast framing. Can be a dedicated chapter mid-video.
🔔 The Action Moment
One chapter should be 60–90s of step-by-step: "Here's exactly what to do right now." Actionable endings are the #1 predictor of subscribe conversion on educational finance content.

SECTION 4 — DATA VISUALISATION ANIMATIONS
═══════════════════════════════════════════

This is the critical layer. Since scripts contain 80+ data points across 9 chapters,
each chapter animation must display data — not just abstract patterns.
Minimum 5–7 verified or estimated data points per chapter must be visually represented.

TRANSITION SPEED PHILOSOPHY — NON-NEGOTIABLE:
Graphics and data sets must transition at professional editing pace — not slow-spread.
The rule is: transition fast, then hold while relevant captions are on screen.

TRANSITION TIMING RULES:
  — Animate IN:  every graphic element enters in 0.25–0.45s (spring or ease-out). NEVER over 0.6s.
  — HOLD:        once fully rendered, the graphic holds at full opacity for the duration of its
                 associated SRT caption block(s). The hold is determined by iconTimes[] — the graphic
                 stays visible until the next icon trigger fires, or until chapter end.
  — Animate OUT: fade or slide out in 0.2–0.3s when the next graphic enters.
  — Sequential elements (bars, timeline nodes, flow boxes): stagger entry by 120–180ms max.
    Do not space stagger beyond 200ms — it reads as broken, not dramatic.
  — Stat slams: spring-in over 18–22 frames at LOOP_FPS. Full opacity by frame 22. Hold until
    the next icon trigger. Do not cycle or fade a stat while its caption is still on screen.
  — Growth curves: draw left-to-right over 0.5s maximum, then hold at full draw.
    Do not time the draw to stretch over the full chapter — draw it, then let it sit.
  — Comparison bars: all bars fill simultaneously over 0.4s. Hold.
  — Timeline nodes: animate in left-to-right with 150ms stagger. Each node snaps in (not floats in).
  — Myth/Reality rows: each row pair animates in with 180ms stagger. Strikethrough fires at prog>0.60
    in 0.2s. Hold.
  — Flow diagram boxes: spring in with 160ms stagger. Connector arrows draw in 0.15s.
  — Hold frames (share-bait): enter in 0.3s, hold for minimum 4 seconds, no animation during hold.

SECTION 4B — VIDEO RENDERING SYSTEMS (NON-NEGOTIABLE)
═══════════════════════════════════════════

These four systems must be present in every VideoGen HTML file without exception.
They are the primary difference between a slideshow and a video.
Omitting any one of them makes the output feel like a presentation, not broadcast content.

──────────────────────────────────────────
4B-1 · COUNT-UP NUMBERS (numStat)
──────────────────────────────────────────

PURPOSE:
  A stat that springs in fully-formed is a reveal. A stat that counts up to its value
  is a performance — the viewer watches the number build and land. This single change
  is responsible for roughly 50% of the "video vs slideshow" perception difference.

RULE: Every numeric stat slam MUST use numStat(), never stat() directly.
  — Integer values: count from 0 → target
  — Dollar values: count from $0 → $target with comma formatting
  — Percentage values: count from 0% → target%
  — Year values (e.g. 1943): count up from a contextually plausible start (e.g. 1900)
  — Text-only stats (FIXABLE, ZERO, ONLY, words): use stat() as normal — no count-up

IMPLEMENTATION — copy this helper once, call everywhere:
  function numStat(target, pfx, sfx, cx, cy, col, sz, cp) {
    // cp = entry progress 0→1 (the same spring value used to scale sz)
    const t = clamp(cp / 0.85, 0, 1);
    const counted = cp >= 0.85 ? target : Math.round(target * Math.pow(t, 0.6));
    const fmtd = pfx + (counted >= 1000 ? counted.toLocaleString() : String(counted)) + sfx;
    stat(fmtd, cx, cy, col, sz);
  }

CALL PATTERN:
  // Always pass the spring entry progress (sp) as the last argument:
  const sp = spring(apR(cs, phaseStart, 0.7));
  const sz = clamp(H * .13, 55, 160) * sp;
  numStat(3167, '$', '', W*.5, H*CY_PH, C.red, sz, sp);

HOLD FRAME numbers:
  The hold-frame build has its own progress variable (hp). Count up through its
  first 85% then hold at final value:
    const _hv = hp >= 0.85 ? target : Math.round(target * Math.pow(clamp(hp/0.85,0,1), 0.6));
    const _hs = '$' + _hv.toLocaleString();
    // Then render ghost + main stat using _hs instead of the hardcoded string.

──────────────────────────────────────────
4B-2 · AMBIENT PARTICLES
──────────────────────────────────────────

PURPOSE:
  The canvas must never be completely motionless. Particles provide the minimal
  constant motion that signals "this is video, not a slide." They are intentionally
  very subtle — opacity 0.024–0.057, radius 1.2–3px. Viewers don't consciously
  notice them; they notice their absence.

RULE: drawParticles() is called in EVERY frame, in both render() and the export loop,
  immediately after drawPaper() and before drawGrid().

IMPLEMENTATION:
  // Deterministic seeds — no Math.random() (breaks frame-accurate export)
  const PARTS = (function() {
    const s = [0.137,0.239,0.341,0.431,0.512,0.619,0.724,0.813,0.907,
               0.178,0.056,0.462,0.881,0.325];
    return s.map((v, i) => ({
      ox:  ((v*7.3   + s[(i+3)%14]*.8) % 1),
      oy:  ((s[(i+5)%14]*5.1 + s[(i+7)%14]*.6) % 0.88),
      r:   1.2 + (i % 4) * .55,
      spd: 0.007 + (i % 5) * .0035,
      ang: (i * 47.11) % (Math.PI * 2),
      al:  0.024 + (i % 4) * .011
    }));
  })();

  function drawParticles() {
    const t = currentSec;
    ctx.save();
    PARTS.forEach(p => {
      const px = ((p.ox + Math.cos(p.ang) * p.spd * t) % 1 + 1) % 1;
      const py = ((p.oy + Math.sin(p.ang) * p.spd * t) % 0.88 + 0.88) % 0.88;
      ctx.beginPath(); ctx.arc(px * W, py * H, p.r, 0, Math.PI * 2);
      ctx.fillStyle = C.ink; ctx.globalAlpha = p.al; ctx.fill();
    });
    ctx.globalAlpha = 1; ctx.restore();

9B · DURATION-ACCURATE EXPORT (Worker-based frame pacing)
──────────────────────────────────────────

ROOT CAUSE OF v1 DURATION BUG: frames were rendered as fast as possible. MediaRecorder
timestamps from wall-clock time, so a 40s chapter rendering in 5s produced a 5s video.
Resolution/FPS changed render speed → every combination produced a different wrong duration.

BACKGROUND-TAB FREEZE BUG (v4 and earlier): The previous fix used main-thread setTimeout
to pace frames to wall-clock time. When the user switched apps, Chrome throttled main-thread
timers to 1 second minimum — the pacing loop stalled, frames stopped submitting, but the
MediaRecorder wall-clock kept running. Result: the exported video had the correct frame
count but wrong (inflated) duration.

CORRECT APPROACH — WEB WORKER TIMER:
  Workers run in a separate thread exempt from Chrome's background-tab timer throttling.
  Worker setInterval fires at the configured interval (~1ms precision) even when the tab
  is not visible. The worker posts 'tick' messages to the main thread, which submits one
  frame per tick. Main-thread message delivery is not throttled (unlike setTimeout).

  STEP 1: Define makeTimerWorker() — creates an inline worker with setInterval:
    function makeTimerWorker() {
      const code = `var t; self.onmessage = function(e) {
        if (e.data.cmd === 'start') {
          t = setInterval(function() { postMessage('tick'); }, e.data.ms);
        } else if (e.data.cmd === 'stop') {
          clearInterval(t);
        }
      };`;
      return new Worker(URL.createObjectURL(new Blob([code], {type:'application/javascript'})));
    }

  STEP 2: Define drawExportFrame(ci) — shared render helper that centralises the exact
  render order (matching render()) for both single and combined export paths:
    function drawExportFrame(ci) {
      drawPaper(); drawParticles();
      const cs0 = currentSec - CHAPTERS[0].start;
      const cs8 = currentSec - CHAPTERS[8].start;
      const isCard = (ci===0 && cs0<2.8) || (ci===lastCi && cs8>=cardStart);
      if (!isCard) drawGrid();
      const p = chP(ci);
      ANIM_FNS[ci](p, currentSec*LOOP_FPS, W, H);
      if (!(ci===0 && cs0<2.8)) drawCenterPhrase(ci, p);
      drawLowerThird(ci); drawVignette(); drawChapterFade(ci);
    }

  STEP 3: Export loop — one frame per worker tick, no setTimeout pacing needed:
    let frameIndex = 0;
    const worker = makeTimerWorker();
    await new Promise(resolve => {
      worker.onmessage = () => {
        if (frameIndex >= totalFrames) { worker.postMessage({cmd:'stop'}); resolve(); return; }
        currentSec = ch.start + frameIndex / fps;
        drawExportFrame(ci);
        track.requestFrame();
        frameIndex++;
        // update progress UI
      };
      worker.postMessage({cmd:'start', ms: 1000/fps});
    });
    mr.stop(); await new Promise(r => mr.onstop = r);

  Result: MediaRecorder wall-clock timestamps match frame submission timing from the
  worker — accurate regardless of whether the tab is visible or not.

FORMATS & SETTINGS:
  — captureStream(0), mr.start(500)
  — WebM VP9 preferred, MP4 fallback
  — Resolutions: 720p / 1080p / 4K  (W = H * 16/9)
  — Frame rates: 30fps / 60fps
  — Bitrates: 4K=80Mbps, 1080p=22Mbps, 720p=9Mbps
  — Separate-chapter filename: W##_Ch[N]_TopicShort_ChShort_[dur]s.ext
  — Combined filename: W##_TopicShort_Ch1-Ch2-...-ChN_Combined_[totalDur]s.ext
  — Show export panel note describing Worker timer and combined vs separate mode
  — Download trigger: anchor click

  1. Define every column as an explicit named constant derived from a left anchor (x0):

       const c3 = x0 + W * 0.47;      // col 3
       const c4 = x0 + W * 0.68;      // col 4 — adjust if 4-col layout

     Single-variable colW causes columns to share the same X coordinate when offsets
     are added in different directions (e.g. colW + colW*0.05 vs colW*2 + colW*0.05).
  2. Use the same named constant for BOTH the header row and every data row of that column.
     Never use c3 for the header and c3 + W*0.02 for the data — they must be identical.
  3. The rightmost column must fit within W - 56px (canvas safe margin).

  4. Highlight rows (e.g. the final / most important row): extend the background rect from

10B · COLOUR VISIBILITY ON PARCHMENT
──────────────────────────────────────────

Problem: teal (#2DB89D), amber (#F5C518), slate (#8BA8B5), and all grays read dim
against the warm parchment background (#E4DDD0 / #DEDAD1) — especially at video
bitrates and on warmer-calibrated screens. Do NOT change the palette.
Use these rendering techniques instead:

RULE 1 — SHADOW HALOS FOR ALL COLOURED TEXT AND LINES:
  For any text drawn in C.teal, C.amber, C.amberD, C.slate, C.grayL, or C.red:
    ctx.shadowColor = 'rgba(255,255,255,0.55)';
    ctx.shadowBlur  = 6;
  Draw the text. Then reset:
    ctx.shadowColor = 'transparent'; ctx.shadowBlur = 0;
  This creates an invisible white glow that lifts the colour off the parchment
  without altering hue. Do not apply to C.ink or C.ink2 (dark text reads fine).

RULE 2 — THIN WHITE OUTLINE FOR LOW-CONTRAST STROKED SHAPES:
  For any strokeStyle using C.teal, C.amberD, C.slate, or C.grayL at lineWidth < 2:
    Draw the shape twice:
      Pass 1: lineWidth + 2.5, strokeStyle = 'rgba(255,255,255,0.50)' (white halo pass)
      Pass 2: original lineWidth, original strokeStyle (colour pass)
  This gives teal/amber lines a crisp visible edge without changing their colour.

RULE 3 — FILL OPACITY FLOOR:
  Any fillStyle using the palette colours must have alpha >= 0.72.
  Never use teal/amber/slate at alpha < 0.72 for text or icons. For backgrounds/overlays
  a lower alpha is fine — but for legibility-critical elements, enforce the floor.

RULE 4 — ICON ACTIVE STATE CONTRAST:
  Active icon border: lineWidth 2.2 (up from 1.8), apply Rule 2 white halo pass first.
  Active icon label: apply Rule 1 shadow halo.
  Active icon interior fill: rgba(45,184,157, 0.16) (slightly denser than default .10).

RULE 5 — DATA LINES AND CURVES:
  drawLine() and dashedLine() for C.teal or C.amber: apply white halo pass (Rule 2)
  before the colour pass. Minimum lineWidth 2.0 for any animated data line.
  Growth curve endpoint circles: add a white filled circle 1.5× the dot radius beneath
  the coloured dot (drawn first) to give it a lifted appearance.

RULE 6 — AMBER STAT TEXT EXCEPTION:
  C.amber (#F5C518) is bright but low-contrast on parchment. For stat() calls using
  C.amber, instead use C.amberD (#C8950A) for the text colour, then apply Rule 1.
  C.amberL (#F9D650) may be used for decorative bar underlays only — never for text.

RULE 7 — COMPARISON BAR FILLS:
  Fill bars with a solid colour at full opacity (alpha 1.0), not semi-transparent.
  Add a 1px white separator line between adjacent bars to prevent colour bleed.

Apply all 7 rules consistently across every draw call in every chapter animation.
These rules produce no visual style change — they only ensure the intended colours
read clearly at any screen brightness and after video compression.

──────────────────────────────────────────
11A · THE OVERLAP PROBLEM (and why it matters)
──────────────────────────────────────────

Each chapter animation is divided into sequential phases, each driven by a time
window returned by phA(cs, start, end, fade). A phase is visible when cs (seconds
into the chapter) is within [start, end].

THE RULE: start of Phase N+1 MUST equal end of Phase N. Zero exceptions.

If Phase 1 ends at cs=8.5 and Phase 2 starts at cs=7.0, both phases are alive
simultaneously for 1.5 seconds. The viewer sees two visuals on screen at once.
This is the overlap bug. It was the primary visual quality problem in v1 and v2.

──────────────────────────────────────────
11B · phA() FUNCTION CONTRACT
──────────────────────────────────────────

phA(cs, start, end, fade) returns alpha 0→1→0 over the window [start, end].
  — fade = the crossfade duration in seconds (default 0.35s). The phase fades IN
    over the first `fade` seconds and fades OUT over the last `fade` seconds.
  — Between fade-in and fade-out, alpha = 1.0 (fully visible).

CORRECT USAGE:
  // Chapter has 4 phases across 40.3 seconds:
  const a0 = phA(cs,  0.0,  8.5, 0.35);   // Phase 1: 0→8.5s
  const a1 = phA(cs,  8.5, 20.0, 0.35);   // Phase 2: 8.5→20.0s   ← starts where P1 ends
  const a2 = phA(cs, 20.0, 31.0, 0.35);   // Phase 3: 20.0→31.0s  ← starts where P2 ends
  const a3 = phA(cs, 31.0, 40.3, 0.35);   // Phase 4: 31.0→40.3s  ← starts where P3 ends

WRONG USAGE (creates overlap):
  const a0 = phA(cs,  0.0, 11.0, 0.35);   // Phase 1 ends at 11.0
  const a1 = phA(cs,  8.5, 20.0, 0.35);   // Phase 2 starts at 8.5 — OVERLAP 8.5→11.0 ✗
